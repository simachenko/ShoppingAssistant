# Quickstart: Smart Product Advisor

This guide proves the feature works end-to-end locally. It does not contain implementation
code — see `tasks.md` (generated by `/speckit-tasks`) for build-out steps, and `data-model.md`
/ `contracts/` for the shapes referenced below.

## Prerequisites

- .NET 10 SDK
- Docker Desktop (or compatible engine) running — required by both .NET Aspire's Postgres
  resource and the docker-compose fallback
- An API key for the configured free-tier LLM provider (see `research.md` §10), set as a
  user-secret or environment variable — never committed
- Seed data: a small fixture set covering at least one category (e.g., "Smartphones") with
  several products, specifications, and matching Pricing offers, so the scenarios below have
  real data to work against

## Option A: Run via .NET Aspire (primary local dev path)

```bash
dotnet run --project src/Aspire/AppHost
```

This starts Postgres, applies EF Core migrations for `catalog`/`pricing`/`advisor` schemas,
starts all five services with service discovery wired up, and opens the Aspire dashboard
(traces, logs, metrics) so you can watch a request fan out across services.

## Option B: Run via Docker Compose (CI-parity fallback)

```bash
docker compose up --build
```

## Validation Scenario 1 — Recommendation from a natural-language need (US1, SC-001, SC-003, SC-004)

1. Open the Blazor app (URL printed by Aspire/compose) and start a new chat.
2. Send: `I need a smartphone with a good camera and a budget of up to 15,000 UAH.`
3. **Expect**: a `recommendation` response (per `contracts/advisor-conversation-api.md`) with
   one or more items, each showing a price at or under 15,000 UAH, at least one matched
   requirement referencing the budget or camera, and at least one trade-off — with no more than
   the one message round trip above (no clarification needed, since category + budget + a
   feature were all supplied).

## Validation Scenario 2 — Clarification before recommending (US1, FR-002, FR-003, SC-005)

1. Start a new chat and send: `I need a good laptop.`
2. **Expect**: a `clarification` response asking specifically about the missing budget (not a
   recommendation, and not more than one question at once).
3. Reply with a budget, e.g. `Up to 30000 UAH`.
4. **Expect**: a `recommendation` response now follows, using the budget just provided.

## Validation Scenario 3 — No match communicated honestly (FR-007, FR-010, edge case)

1. Start a new chat and send a request with a budget below every seeded product's price in
   that category, e.g. `I need a smartphone for under 500 UAH.`
2. **Expect**: `items: []` and a non-null `unmetConstraintExplanation` — never a product priced
   above 500 UAH presented as a match.

## Validation Scenario 4 — Comparison with consistent criteria (US2, FR-006, SC-002)

1. In an existing session (or a new one), send: `Compare [Product A] and [Product B].` using
   two seeded product names.
2. **Expect**: a `comparison` response whose `criteria` array is identical for both rows and in
   the same order (per `Category.ComparableAttributeKeys` in `data-model.md`), and whose `rows`
   each carry a `rating` and `deltasVsBest` — all produced by the `compare_products` tool (see
   `contracts/advisor-mcp-tools.md`), never computed by the chat text itself. Re-run the same
   comparison a second time and confirm `rating`/`deltasVsBest` are identical both times.

## Validation Scenario 5 — Verified fact lookup and "not found" honesty (US3, FR-005, FR-012)

1. Send: `Is [Product A] in stock and what does it cost?`
2. **Expect**: a response with `priceVerified: true`/`availabilityVerified: true` sourced from
   the seeded Pricing offer, including its `asOf` freshness if asked follow-up.
3. Send: `What's the price of Totally Fake Product 9000?`
4. **Expect**: a clear "not found" answer — never an invented price or spec.

## Validation Scenario 6 — Partial failure is honest, not a crash (Constitution Principle V)

1. Stop the Pricing service container only (`docker compose stop pricing-availability-api`, or
   pause its Aspire resource).
2. Repeat Validation Scenario 1.
3. **Expect**: a `200 recommendation` response still returns, but items show
   `priceVerified: false` / `availabilityVerified: false` instead of a stale or fabricated
   price, and the chat text explicitly says pricing couldn't be confirmed right now.
4. Restart the Pricing service before continuing.

## Running the automated test suite

```bash
dotnet test                     # unit + application + contract tests (fast, no containers needed beyond Testcontainers-managed Postgres)
docker compose -f docker-compose.yml -f docker-compose.ci.yml up --build --abort-on-container-exit  # EndToEnd.Tests cross-service scenarios
```

All of the scenarios above have a corresponding automated test in
`tests/EndToEnd.Tests`; this manual walkthrough is for demoing the feature, not a substitute
for the automated suite gating merges (constitution Principle III).
